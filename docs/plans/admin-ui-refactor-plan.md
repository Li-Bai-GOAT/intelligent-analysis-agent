# DataAgent 管理后台前端重构方案

## 1. 文档目的

本文用于指导 DataAgent 系统管理员后台的前端重构，并同步约束必要的后端接口补充、权限控制、测试和验收流程。

本轮仅制定方案，不修改管理端业务代码。实施时应保持现有对话页行为稳定，避免后台重构影响 KunCode、沙箱终端、文件面板和普通用户对话流程。

## 2. 当前状态结论

### 2.1 已有基础

- React 前端已经具备 `auth`、`app`、`admin` 三种页面状态。
- 管理员登录后可以从右上角菜单进入管理后台，也可以返回对话页。
- `page === 'admin'` 目前只显示“开发中”占位内容，没有正式管理界面。
- 前端已有 Button、Input、Modal、Switch、Tabs、ErrorBoundary 等基础组件。
- 后端已经提供知识库、系统提示词、Agent、Skill、MCP 的主要管理接口。
- 后端已有 `/ready` 健康检查接口，可报告 PostgreSQL、Redis、Sandbox、Milvus 和 Embedding 状态。
- Skill 已支持 ZIP 上传、文件树浏览、文件内容读取和 Agent 权限配置。
- Sandbox 已支持把 Agent、Skill、MCP 配置注入指定容器。

### 2.2 主要缺口

| 级别 | 缺口 | 影响 |
| --- | --- | --- |
| P0 | 管理后台仍为占位页 | 管理员无法通过 UI 使用已有管理能力 |
| P0 | 管理接口大量使用 `Record<string, unknown>` | 字段错误只能在运行时暴露，容易产生前后端契约偏差 |
| P0 | Knowledge、Agent、Skill、MCP 的部分前端类型与后端模型不一致 | UUID、tools、command、`mcp_type` 等字段可能提交或展示错误 |
| P0 | 后台路由只有前端页面状态，没有独立的后台权限边界组件 | 权限失效、刷新和接口 401/403 时缺少统一处理 |
| P1 | 没有后台信息架构、列表状态、表单状态和错误反馈规范 | 后续页面容易重复实现并产生交互冲突 |
| P1 | 注入接口要求调用方直接提供 `container_id`，但没有活动沙箱列表接口 | 管理员难以可靠选择目标容器 |
| P1 | MCP 缺少连接测试接口 | 保存配置不等于配置可用，故障发现过晚 |
| P1 | 系统提示词没有版本、差异比较和回滚接口 | 误修改后恢复成本高 |
| P1 | 管理操作没有审计日志 | 无法追踪配置由谁、在何时、如何修改 |
| P2 | 没有用户管理接口 | 暂时无法实现账号禁用、角色调整和密码重置 UI |
| P2 | 没有独立的后台总览聚合接口 | 总览页只能并行请求多个列表接口，效率和一致性一般 |

## 3. 重构目标

1. 为管理员提供完整、清晰、可重复操作的管理工作台。
2. 第一阶段优先覆盖后端已经具备的知识库、提示词、Agent、Skill 和 MCP 管理能力。
3. 所有管理数据使用明确的 TypeScript 类型和统一 API 错误模型。
4. 所有写操作具备保存中、成功、失败、未保存和危险操作确认状态。
5. 普通用户不能进入后台，也不能直接调用后台接口。
6. 后台保持紧凑、安静、面向运维工作的视觉风格，不复制对话页终端布局。
7. 为后续用户管理、审计日志、运行时部署和系统配置预留稳定扩展点。

## 4. 非目标

- 不在后台中重新实现对话页、终端或文件面板。
- 不允许管理员直接在浏览器中编辑服务器环境变量或查看完整 API Key。
- 不为尚无后端数据来源的指标制作模拟图表。
- 不在首期实现复杂 RBAC；首期只区分管理员与普通用户。
- 不在首期实现数据库、Redis 或 Milvus 的通用 Web 运维控制台。

## 5. 推荐信息架构

后台采用固定管理壳层，桌面端使用紧凑侧栏，窄屏折叠为抽屉。顶部区域保留页面标题、服务状态摘要、返回对话页和账号菜单。

### 5.1 一级导航

| 导航 | 首期范围 | 数据来源 |
| --- | --- | --- |
| 总览 | 服务健康状态、配置数量、最近更新时间 | `/ready` 与现有列表接口；后续改为聚合接口 |
| 知识库 | 查询、分页、新增、编辑、删除 | Knowledge API |
| 系统提示词 | 编辑、保存、未保存提醒 | System Prompt API |
| 沙箱配置 | Agent、Skill、MCP 三个二级页签 | Sandbox API |
| 运行时部署 | 选择活动沙箱并注入配置 | 现有 Injection API；需新增活动沙箱列表 API |
| 审计日志 | 管理变更记录 | 第二阶段新增后端接口 |
| 用户管理 | 用户、角色、禁用、密码重置 | 第二阶段新增后端接口 |

### 5.2 页面布局规则

- 列表页统一使用“标题和主操作 + 筛选栏 + 数据表格 + 分页”。
- 编辑采用独立页面或宽侧栏，不把复杂表单塞入小型弹窗。
- 删除、覆盖和运行时注入必须二次确认。
- 页面区块使用无边框布局或细分隔线；卡片只用于独立状态项和重复实体。
- 桌面端优先保证信息密度，移动端保证字段不重叠并允许表格切换为纵向条目。
- 图标按钮使用现有图标库并提供 tooltip，不使用文字胶囊代替通用图标。

## 6. 功能模块设计

### 6.1 总览

首期展示：

- PostgreSQL、Redis、Sandbox、Milvus、Embedding 健康状态。
- Knowledge、Agent、Skill、MCP 的启用数和总数。
- 系统提示词最近更新时间。
- 健康检查失败时提供错误摘要和重新检测按钮。

约束：

- 不显示无真实数据支持的调用量、成功率和 Token 趋势。
- 首期可并行调用现有接口聚合数据，但要设置独立超时和局部失败状态。
- 第二阶段新增 `/api/admin/overview`，由后端一次性返回健康状态和计数。

### 6.2 知识库

首期能力：

- 按名称或内容查询、分类筛选、分页。
- 新增、编辑、查看和删除知识条目。
- 表单校验、重复名称提示、删除确认。
- 明确显示创建时间和更新时间。

后端补充建议：

- 管理列表和详情接口强制管理员鉴权，避免知识内容被普通 HTTP 请求直接枚举。
- 增加服务端搜索和分类筛选参数。
- 增加向量索引状态、失败原因和重新索引接口。
- 将面向 Agent 的知识检索与后台 CRUD 接口拆分，权限职责保持清晰。

### 6.3 系统提示词

首期能力：

- Markdown 或纯文本编辑器。
- 字符数、最后更新时间和当前提示词名称。
- 未保存修改提示、保存确认、保存失败后保留编辑内容。
- 刷新或离开页面前进行脏数据拦截。

第二阶段能力：

- 版本历史、版本备注、差异比较和一键回滚。
- 发布前预览和基础变量校验。

版本与回滚必须先增加后端持久化模型，不能只在浏览器本地保存历史。

### 6.4 Agent 管理

首期能力：

- 列表展示名称、模式、启用状态、温度、最大步骤数和更新时间。
- 新增、查看、编辑、启用或停用、删除。
- 编辑字段覆盖 `name`、`description`、`mode`、`tools`、`permission`、`temperature`、`max_steps`、`hidden`、`content` 和 `enabled`。
- `mode` 使用单选或分段控件，布尔值使用 Switch，数值使用范围明确的输入控件。
- `tools` 和 `permission` 优先使用结构化选择器；仅提供受校验的高级 JSON 编辑模式。
- 后端不允许修改 Agent 名称时，编辑页应将名称设为只读并说明创建后不可变。

校验要求：

- 温度和最大步骤数必须符合后端范围。
- Agent 名称格式和唯一性在前后端同时校验。
- 工具引用必须来自可用工具集合，禁止静默保存未知工具。

### 6.5 Skill 管理

首期能力：

- ZIP 上传，显示文件名、大小、上传进度和服务端解析错误。
- 列表展示名称、描述、启用状态和更新时间。
- 查看 Skill 文件树，并以只读方式预览文件内容。
- 启用或停用、删除。
- 按 Agent 配置 `allow`、`deny`、`ask` 权限矩阵。

交互约束：

- 上传后自动刷新列表并定位新 Skill。
- 文件树加载失败不能阻断 Skill 基本信息展示。
- 权限矩阵保存失败时恢复到服务端状态并显示具体失败项。
- 文件路径必须完全使用后端返回值，不由前端拼接宿主机路径。

### 6.6 MCP 管理

首期能力：

- 列表、新增、查看、编辑、启用或停用、删除。
- `local` 与 `remote` 使用分段控件切换，并显示不同字段。
- Local MCP 编辑命令数组和环境变量。
- Remote MCP 编辑 URL 和请求头。
- Header 和环境变量使用键值编辑器，敏感值默认遮蔽。

后端补充建议：

- 新增 MCP 连接测试接口，返回阶段、耗时和脱敏错误。
- 增加敏感字段标记，读取配置时不返回密钥原文。
- 明确空值语义，避免更新时意外覆盖已有密钥。
- 服务端验证远程 URL 协议、本地命令结构和字段互斥规则。

### 6.7 运行时部署

目标是让管理员把数据库中的配置同步到明确的活动沙箱，而不是手动输入不透明的容器 ID。

首期流程：

1. 从活动沙箱列表选择目标会话或容器。
2. 选择注入范围：全部、Agent、Skill 或 MCP。
3. 展示待部署配置摘要。
4. 二次确认后调用现有注入接口。
5. 展示每一类配置的成功数、跳过数和失败原因。

实施前必须增加活动沙箱列表接口，至少返回：

- `container_id`
- 关联用户与对话 ID
- 创建时间和最后活动时间
- 当前状态
- 已注入配置版本或时间

禁止把自由输入 `container_id` 作为主要交互；可在高级调试模式中保留，并要求显式确认。

### 6.8 用户管理与审计日志

这两个模块属于第二阶段，因为当前后端没有完整接口。

用户管理建议支持：

- 用户列表、角色筛选和状态筛选。
- 启用或禁用账号。
- 管理员发起密码重置，不展示现有密码。
- 管理员角色调整，并阻止删除或降级最后一个管理员。

审计日志建议记录：

- 操作人、资源类型、资源 ID、动作、时间和结果。
- 修改前后差异的脱敏摘要。
- 请求关联 ID 和失败原因。
- Knowledge、Prompt、Agent、Skill、MCP、注入、用户权限等关键操作。

### 7.2 组件边界

- `App.tsx` 只负责认证状态和顶层页面切换，不承载具体后台页面逻辑。
- `AdminApp.tsx` 负责后台权限门禁、后台路由和错误边界。
- 每个资源页独立管理查询参数和编辑状态，禁止集中到一个超大组件。
- 通用表格、确认框、状态标签和键值编辑器只抽取一次。
- 对话页状态与后台状态分开，避免切换页面时污染当前会话和终端状态。

### 7.3 状态管理

- URL 或后台路由保存当前模块、分页、筛选和选中资源，支持刷新恢复。
- Zustand 只保存后台 UI 偏好和跨页面状态，不缓存所有服务端实体。
- 服务端数据使用统一查询层；若不引入新依赖，则封装 AbortController、加载状态、重试和失效刷新。
- 表单状态留在页面或表单组件内，提交成功后再更新服务端数据缓存。
- 只对启用开关等可回滚操作采用乐观更新；创建、删除、提示词保存和注入使用服务端确认结果。

### 7.4 类型与 API 客户端

第一项代码任务是建立真实接口契约，替换后台相关的 `Record<string, unknown>`。

必须修正：

- Knowledge ID 使用 UUID 字符串，而不是 number。
- Agent `tools` 和 `permission` 按后端字典结构建模，而不是 `string[]`。
- MCP 使用 `mcp_type`，取值为 `local | remote`。
- MCP `command` 使用字符串数组。
- 分页响应统一为 `{ items, total, limit, offset }`。
- 创建、更新和详情响应分别定义类型，不复用含服务端只读字段的提交对象。
- 统一解析 401、403、404、409、422 和 500 错误，并保留后端错误详情。

建议为后台 API 设置独立导出入口，避免继续扩大当前通用 `client.ts`。

## 8. 后端配套改造

### 8.1 首期必须完成

- 校正并记录所有后台 API 请求和响应模型。
- Knowledge 管理列表和详情增加管理员鉴权，或拆分公共检索接口与后台管理接口。
- 新增活动沙箱列表接口，支持运行时部署页面。
- 所有后台接口统一返回可展示的错误码和错误消息。
- 对敏感 MCP Header 和环境变量进行服务端脱敏。
- 为管理写操作增加审计记录基础能力。

### 8.2 第二阶段建议

- `GET /api/admin/overview`：健康状态、实体计数和最近变更。
- `GET /api/admin/sandboxes`：活动沙箱列表。
- `POST /api/admin/mcps/{id}/test`：MCP 连接测试。
- Prompt 版本列表、差异和回滚接口。
- Knowledge 索引状态和重新索引接口。
- 用户列表、禁用、角色调整和密码重置接口。
- 审计日志查询与筛选接口。

具体路径应遵循项目现有路由前缀，实施前通过 OpenAPI 或路由测试固定契约。

## 9. 权限与安全要求

- 前端门禁只用于交互，所有权限必须由后端再次验证。
- 普通用户访问后台页面时跳回对话页并显示权限提示。
- 401 清理失效登录状态；403 保留登录并显示无权限状态。
- API Key、Authorization Header、密码和环境变量敏感值不得写入日志、Toast 或审计差异。
- 敏感字段读取时返回遮蔽值和 `has_value` 标记，更新时支持“保持不变”语义。
- Markdown、日志和文件预览必须防止 XSS，不直接执行 HTML 或脚本。
- Skill 文件浏览继续依赖后端路径校验，前端不得允许任意路径读取。
- 删除、回滚、注入和权限变更必须记录操作人和结果。
- 管理表单不接受未经结构校验的任意 JSON 作为默认编辑方式。

## 10. 统一交互状态

每个管理页面必须实现以下状态：

- 首次加载骨架或进度状态。
- 空数据状态，并提供与当前模块一致的主操作。
- 局部请求失败状态和重试按钮。
- 保存中禁止重复提交。
- 成功反馈包含资源名称和结果，不只显示“成功”。
- 422 显示到具体字段，409 显示冲突原因。
- 删除和注入操作显示明确影响范围。
- 表单有未保存修改时，切换模块、返回对话页和刷新浏览器均提示确认。
- 长列表请求支持取消，快速切换筛选时旧响应不能覆盖新结果。

## 11. 分阶段实施顺序

### 阶段 0：契约基线与回归保护

- 整理后台接口清单和请求响应样例。
- 修正 TypeScript 后台类型。
- 补充管理员和普通用户权限接口测试。
- 建立对话页关键流程回归测试，防止后台改造影响现有功能。

验收：前后端字段一致，类型检查通过，现有对话和 KunCode 流程不回归。

### 阶段 1：管理壳层与总览

- 实现 `AdminApp`、后台导航、权限门禁和响应式布局。
- 接入 `/ready` 和现有列表计数。
- 统一加载、错误、空状态和通知机制。
- 支持后台刷新后恢复当前模块。

验收：管理员可进入和离开后台，普通用户不可访问，依赖故障能准确显示。

### 阶段 2：知识库与系统提示词

- 完成 Knowledge CRUD、筛选、分页和删除确认。
- 完成 Prompt 编辑、保存和未保存拦截。
- 收紧 Knowledge 管理接口权限。

验收：所有写操作可恢复、错误可定位，提示词刷新后与服务端一致。

### 阶段 3：Agent 与 Skill

- 完成 Agent 列表、结构化编辑、启停和删除。
- 完成 Skill 上传、列表、文件预览、权限矩阵、启停和删除。
- 对上传和权限保存补充细粒度错误反馈。

验收：数据库配置与 UI 展示一致，Skill 文件和权限可以完整查看及修改。

### 阶段 4：MCP 与运行时部署

- 完成 Local/Remote MCP 表单及敏感字段处理。
- 增加 MCP 测试接口和测试结果展示。
- 增加活动沙箱列表接口。
- 完成按范围注入配置和结果汇总。

验收：管理员无需手输容器 ID 即可完成配置部署，失败项有明确原因，密钥不泄漏。

### 阶段 5：审计、用户管理与提示词版本

- 增加审计日志模型、接口和查询页面。
- 增加用户管理后端能力和页面。
- 增加提示词版本、差异和回滚。

验收：关键变更可追踪、可恢复，账号管理具备服务端权限保护。

### 阶段 6：完整验收与发布

## 12. 测试方案

### 12.1 前端测试

- 管理页面权限门禁和 401/403 行为。
- 类型化 API 的成功、失败、取消和并发响应。
- 表单校验、脏状态、重复提交和删除确认。
- Agent 工具与权限结构化编辑。
- Skill 上传、文件树、内容预览和权限矩阵。
- MCP Local/Remote 字段互斥和敏感值保持不变。
- 注入结果的成功、部分成功和失败状态。

### 12.2 后端测试

- 每个管理端点对管理员成功、普通用户 403、未登录 401。
- CRUD 字段边界、UUID、唯一性和不存在资源。
- Skill ZIP 与路径安全。
- MCP 敏感字段脱敏与连接测试超时。
- Sandbox 列表只返回允许管理员查看的元数据。
- 注入接口的容器状态检查和部分失败返回。
- 审计日志不记录敏感明文。

### 12.3 端到端测试

1. 管理员登录并进入后台，再返回对话页。
2. 普通用户无法访问后台及后台 API。
3. Knowledge 新增、编辑、查询和删除。
4. Prompt 修改、未保存离开提示、保存和重新加载。
5. Agent 新增、编辑、启停和删除。
6. Skill 上传、文件预览、权限修改和删除。
7. Local 与 Remote MCP 创建、测试、编辑和停用。
8. 选择活动沙箱并分别注入 Agent、Skill、MCP 和全部配置。
9. 依赖服务异常时总览准确降级，其他可用页面仍能操作。
10. 后台变更后，新对话和 KunCode 调用仍能正常运行。

## 13. 完成标准

## 14. 推荐提交拆分

为降低评审和回滚风险，建议按以下顺序独立提交：

1. `refactor(admin): define typed admin api contracts`
2. `feat(admin): add admin shell and authorization guard`
3. `feat(admin): add overview and service health`
4. `feat(admin): add knowledge and prompt management`
5. `feat(admin): add agent management`
6. `feat(admin): add skill management`
7. `feat(admin): add mcp management and secret handling`
8. `feat(admin): add runtime configuration deployment`
9. `feat(admin): add audit and user management`
10. `test(admin): add admin integration and e2e coverage`

每个提交都应保持项目可构建、现有测试可运行，不把所有后台页面堆积到一次大提交中。

## 15. 实施优先级结论

最推荐的实施路径是：先完成类型契约、后台壳层和现有 API 对应页面，再扩展活动沙箱、MCP 测试、审计和用户管理后端能力。

这样可以尽快让管理员使用已经存在的功能，同时避免为了展示完整后台而先制造没有真实后端支持的按钮、指标或配置项。首个可交付版本应以 Knowledge、Prompt、Agent、Skill、MCP 管理可用且权限可靠为边界；运行时部署和治理能力随后补齐。
