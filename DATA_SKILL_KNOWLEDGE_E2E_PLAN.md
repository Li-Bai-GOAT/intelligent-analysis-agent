# DataAgent 数据分析 Skill / 知识库端到端执行计划

## 1. 目标

在不改动现有密钥、数据卷和无关用户数据的前提下，补充一组可验证的数据分析知识与 Skill，并通过管理后台完成创建、上传、启用、授权，再从普通对话页面向智能体发起真实任务，确认以下链路完整可用：

```text
管理后台录入/上传
  -> PostgreSQL 元数据与正文
  -> Milvus 向量索引 / Skill 本地目录
  -> SandboxInjectionService 注入运行容器
  -> Agent 权限生效
  -> 对话 SSE 事件与工具调用
  -> 沙箱执行
  -> 回复、结果文件、预览与下载
```

本文档先定义执行方案与验收标准。正式执行时产生的 Skill 包、测试数据、接口记录、截图和缺陷报告另存到 `artifacts/e2e_data_skills/`，不提交运行日志、用户上传文件或敏感配置。

## 2. 已确认的项目基线

- 运行环境为 Windows 本地主机 + Docker Desktop；生产前端由 FastAPI `:8090` 提供。
- 管理后台已有知识库 CRUD、Skill ZIP 上传、启停、文件查看和 Agent 权限配置页面。
- 知识库正文以 PostgreSQL 为事实来源，Milvus 是可重建检索索引。
- Skill ZIP 必须包含带 YAML frontmatter 的 `SKILL.md`，至少提供唯一的 `name` 和 `description`。
- Skill 启用后仍需配置目标 Agent 权限，并在会话沙箱创建/复用时完成注入。
- Agent 内置知识检索只接受四类：`计算公式`、`概念定义`、`分析方法`、`分析流程`。
- 仓库已有 `xlsx`、`data-summarizer`、`correlation-analysis`、`clustering-analysis`、`contribution-analysis` 等本地 Skill 目录；正式执行前必须核对它们是否已有后台记录，不能仅凭目录存在判定为已上传。
- 2026-07-15 的只读基线检查中，`GET /ready` 返回 PostgreSQL、Redis、Sandbox、Milvus、Embedding 全部为 `ok`。本机 PowerShell 执行策略会拦截直接运行 `scripts/start.ps1`，后续状态检查应使用允许的显式执行方式或直接健康接口，不修改系统级执行策略。

## 3. 内容来源与许可证策略

优先自行编写两个窄而可测的 Skill，不下载来源和许可证不清晰的压缩包。实现原则参考以下资料：

- [pandas 用户指南](https://pandas.pydata.org/docs/user_guide/)：缺失值、重复标签、文本、类型和表格整形。
- [NumPy 统计函数](https://numpy.org/doc/stable/reference/routines.statistics.html)：均值、中位数、分位数和加权平均。
- [OpenAI Skill 结构参考](https://github.com/openai/skills/blob/main/skills/.system/skill-creator/SKILL.md)：`SKILL.md` 与 `scripts/`、`references/` 的组织方式。
- [Anthropic Skills 示例仓库](https://github.com/anthropics/skills/blob/main/README.md)：只参考通用目录模式；带专有许可证的现成能力不复制进新增包。

每条知识库内容记录来源 URL、适用口径、边界条件和测试例。涉及业务口径的公式明确标注“示例口径”，避免把通用示例误当成企业正式财务定义。

## 4. 拟新增资产

### 4.1 Skill A：`tabular-data-cleaning`

用途：对 CSV/XLSX 表格执行可审计的数据质量检查和清洗，输出原始数据不覆盖、清洗结果和质量报告。

计划结构：

```text
tabular-data-cleaning/
  SKILL.md
  scripts/profile_and_clean.py
  references/cleaning_rules.md
```

核心规则：

- 识别空值、重复行、重复业务键、空白字符、混合类型、非法日期和异常值候选。
- ID/订单号等标识列保持字符串，避免丢失前导零。
- 缺失值处理必须按列给出理由；默认不静默填充、不静默删除。
- 同时输出 `cleaned_data.csv` 与 `data_quality_report.md`，报告包含处理前后行数、规则命中数、未解决问题和可复核样例。
- 保留输入文件，所有输出写入 `/workspace` 下的新文件。

### 4.2 Skill B：`business-metric-formulas`

用途：按照明确口径计算常用经营指标并做分母、聚合和勾稽校验；适用于 CSV/XLSX 数据与文字口径问答。

计划结构：

```text
business-metric-formulas/
  SKILL.md
  scripts/calculate_metrics.py
  references/formula_contracts.md
```

核心规则：

- 覆盖增长率、毛利额、毛利率、客单价、转化率、加权平均和结构占比。
- 比率类指标优先使用“分子汇总 / 分母汇总”，禁止无说明地平均各行比率。
- 对零分母、缺失口径、单位不一致和负数场景返回显式状态，不伪造数值。
- 输出公式、输入字段映射、中间汇总、最终结果和校验差异。
- 当交付物是 Excel 时复用已有 `xlsx` Skill，保留动态公式并检查 `#REF!`、`#DIV/0!`、`#VALUE!`、`#N/A`、`#NAME?`。

### 4.3 知识库条目

计划新增 8 条带统一测试前缀的知识，便于检索与清理：

| 标题 | 分类 | 关键内容 |
| --- | --- | --- |
| `[E2E] 缺失值处理决策表` | 分析流程 | 删除、填充、保留和人工确认的选择条件 |
| `[E2E] 重复记录与业务主键` | 概念定义 | 完全重复与业务键重复的区别 |
| `[E2E] 字段类型标准化` | 分析方法 | ID、金额、日期、百分比的解析规则 |
| `[E2E] IQR 异常值候选规则` | 分析方法 | 候选识别公式及“不等于错误值”的边界 |
| `[E2E] 加权平均公式` | 计算公式 | `sum(value * weight) / sum(weight)` 与零权重处理 |
| `[E2E] 同比与环比增长率` | 计算公式 | 基期、符号、零分母和百分比展示 |
| `[E2E] 毛利额与毛利率` | 计算公式 | 收入、成本、退货口径与汇总层级 |
| `[E2E] 比率指标聚合原则` | 分析流程 | 汇总比率、行比率平均及勾稽检查 |

## 5. 测试数据

生成一份无真实业务信息的 `e2e_dirty_sales.csv`，使用固定值保证结果可复算。至少包含：

- 带前导零的订单号；
- 前后空格和大小写不一致的地区；
- 两种日期格式及一个非法日期；
- 金额字符串、空金额和负数退货；
- 一条完全重复记录和一组重复业务键；
- 一个明显极端值；
- 转化分母为零的记录；
- 可精确验证的收入、成本、访问量和订单量合计。

同时生成 `expected_results.json`，保存清洗后行数、重复数、有效金额合计、加权平均、毛利率和零分母记录数，供脚本测试和人工验收共用。

## 6. 执行阶段

### 阶段 0：保护现场与健康检查

1. 记录 `git status --short`，不覆盖用户现有改动。
2. 检查 `/live`、`/ready`、Milvus、Embedding 和 Sandbox Manager。
3. 记录后台知识、Skill、Agent 的现有数量和名称；不输出凭据。
4. 确认测试 Agent。优先使用现有可见主 Agent，不新建永久 Agent。
5. 建立测试资产清单，所有名称使用 `[E2E]` 或 `e2e-` 前缀。

### 阶段 1：编写与本地验证资产

1. 编写两个原创 Skill 目录及脚本。
2. 用固定测试数据运行脚本级正向、边界和失败用例。
3. 检查输出不覆盖输入、路径位于工作区、错误信息不含敏感配置。
4. 校验 YAML frontmatter、Skill 名称唯一性、ZIP 内路径和 UTF-8 编码。
5. 分别打包为后台可上传 ZIP；ZIP 可包含单一顶层目录。

### 阶段 2：通过管理后台录入

所有正式录入均从浏览器 UI 完成，以覆盖重构后的前端页面：

1. 管理员登录，进入“知识库”，逐条新增 8 条知识。
2. 验证成功提示、列表分页/分类筛选、编辑后刷新和内容持久化。
3. 进入“Skill 管理”，上传两个 ZIP。
4. 打开 Skill 详情，检查文件树和文本预览。
5. 启用 Skill，为目标 Agent 设置 `允许`，保存后重新打开确认持久化。
6. 记录 UI 截图和对应网络请求状态；不截图或导出令牌。

### 阶段 3：后端与容器一致性核验

1. 通过受认证 API 确认 PostgreSQL 可读取新增知识与 Skill 元数据。
2. 使用独特查询词验证 8 条知识至少各有一次可命中的向量检索。
3. 创建新的测试会话，触发沙箱创建和配置注入。
4. 核对容器中两个 Skill 的 `SKILL.md`、脚本、参考文件均存在。
5. 核对目标 Agent 的 `permission.skill` 为 `allow`，且禁用/未授权 Skill 不应被调用。

### 阶段 4：普通用户对话端到端验证

在普通对话页上传 `e2e_dirty_sales.csv`，依次执行：

1. **知识检索题**：询问加权平均与零分母规则，要求引用知识口径后给一个数值例子。
2. **数据清洗题**：要求检查并清洗上传文件，明确调用 `tabular-data-cleaning`，输出清洗表和质量报告。
3. **公式计算题**：要求按整体口径计算毛利率、转化率和加权平均，明确调用 `business-metric-formulas`。
4. **组合题**：先清洗，再按地区计算指标并生成 Excel，验证新增 Skill 与已有 `xlsx` Skill 的协作。
5. **负向权限题**：临时将一个新增 Skill 设为 `deny` 或停用，在新会话中确认不可调用；恢复后再确认可用。
6. **边界题**：针对零分母、非法日期、缺失金额提问，确认系统显式报告问题而不是静默给出错误值。

每个任务检查 SSE 是否包含稳定的 `tool_id`、合理的 `tool_call/tool_result/status` 顺序，失败路径是否发送 `error` 后正确结束，最终是否总有 `end`，页面是否停止加载且终端状态不残留 `running`。

### 阶段 5：文件交付验证

1. 右侧“文件”页能看到本轮生成的 CSV、Markdown 和 XLSX。
2. 文件路径为相对 `/workspace` 路径，不暴露容器绝对路径。
3. 逐个预览或下载并与 `expected_results.json` 对比。
4. Excel 公式存在且计算值正确，无常见公式错误。
5. 对话最终回复明确列出结果和文件，而不是仅在终端输出路径。

### 阶段 6：回归、缺陷处理与清理

1. 运行后端关键回归测试、前端 lint/build 和 `git diff --check`。
2. 若发现缺陷，先保存最小复现、请求/事件序列和截图，再定位到 UI、API、双写、注入、权限、SSE 或文件服务层。
3. 仅做与本次链路直接相关的最小修复；每个修复补充自动化回归，并重新执行受影响的端到端用例。
4. 删除 `[E2E]` 知识、测试会话和测试生成文件。新增 Skill 是否保留由交付结果决定；默认保留已验收的两个正式 Skill，删除临时 ZIP 和测试数据。
5. 再次检查 `/ready`，确认未影响原有服务和其他 Docker 项目。

## 9. 风险与控制

- **双写部分成功**：每次知识操作同时验证 PostgreSQL 与 Milvus，不把 UI 成功提示当作唯一依据。
- **旧会话复用旧配置**：权限或启停变更后使用新会话/新沙箱复测。
- **第三方 Skill 许可证**：不直接重新分发许可证不明或专有包；新增能力原创并保留来源说明。
- **ZIP 路径与体积风险**：上传前检查路径穿越、绝对路径、软链接、异常压缩比和不必要的大文件。
- **指标口径歧义**：知识和 Skill 都输出公式、字段映射、聚合层级与异常处理规则。
- **污染真实数据**：统一测试前缀、记录创建 ID、按 ID 清理，不批量删除现有条目或数据卷。
- **敏感信息泄露**：证据只保留状态、ID 和脱敏摘要，不记录 JWT、API Key、密码或完整授权头。
